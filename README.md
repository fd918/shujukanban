# 美团联盟活动看板

这是一个静态活动看板，用于查看美团联盟活动的核销金额、档位缺口、预计奖励，并支持手动覆盖每日金额。

## 使用方式

优先直接打开 `index.html`，这是统一入口，左侧可以切换看板，也可以点击左上角按钮收起侧边栏，让看板展示区域最大化：

- “激励活动看板”：对应 `meituan-dashboard-preview.html`。
- “用户内部看板”：本地打开时由统一入口嵌入本机实时服务 `http://127.0.0.1:8791/business-user-dashboard-prototype.html?embed=1`，公网打开时由统一入口嵌入加密公开快照。
- “重点用户看板”：统一入口左侧独立入口，直接进入跨平台、跨业务的关注用户页面；与用户内部看板共用数据和缓存。
- “开放平台毛利”：仅在本机入口显示，读取蒋敏工作日邮件中的 `开放平台毛利表2026.xlsx`，展示累计毛利、月度走势、利润结构、经营底盘、项目贡献与异常日期。经营数据不会进入公网发布文件。

日常只使用统一入口 `index.html`；单独访问 `business-user-dashboard-prototype.html` 会自动跳回统一入口的“用户内部看板”。
其中“用户内部看板”要显示实时数据，需要先确认“业务用户看板服务”已经启动。
公网打开统一入口时，会先要求输入公网看板访问密码；密码与用户内部看板公网快照使用同一套钥匙串配置。
公网入口会在内嵌看板完成数据读取后再撤掉加载提示；手机网络超时会显示“重新加载”，已保存的旧凭证失效时会自动清除并回到密码页，避免空白页面或反复打开。

## 刷新接口数据

推荐直接在本地看板里更新美团请求头：

1. 打开本地看板。
2. 点击右上角“设置”。
3. 在“美团接口请求标头”里粘贴 Chrome F12 复制出的 `pcActivityData` 完整请求标头。
4. 点击“保存请求头”。
5. 回到页面右上角点击“刷新”验证接口是否恢复。

请求头会写入本机文件：

```text
meituan-request-headers.txt
```

该文件已加入 `.gitignore`，不会提交到公开仓库。

命令行刷新仍然可用：

1. 复制 `.env.example` 为 `.env`。
2. 在 `.env` 中填写请求标头文件位置。
3. 在项目根目录执行：

```bash
node scripts/refresh-meituan-data.mjs 1199
```

脚本会全量刷新接口返回的每日数据，并保留看板里手动输入的覆盖值。

当前推荐直接使用“后台常驻刷新”里的 `launchd` 服务；它会同时提供自动刷新和页面“刷新”按钮需要的本机接口。

## 后台常驻刷新

后台服务脚本是：

```bash
node scripts/background-refresh-service.mjs
```

该服务同时做两件事：

- 每隔一段时间自动刷新已开启自动刷新的活动，并记录小时快照。
- 提供页面“刷新”按钮需要的本机接口：`http://127.0.0.1:8765/refresh`。
- 每天 8:00-22:00 每隔 2 小时自动同步一次活动列表，发现新活动后写入看板活动列表。

配置文件在 `data/watch-config.json`：

- `intervalMinutes`：刷新间隔，默认 30 分钟。
- `primaryActivityId`：默认展示的活动 ID。看板打开或刷新后会优先展示这个活动。
- `activityIds`：开启自动刷新的活动 ID 列表。
- `autoPush`：是否刷新后自动提交并推送到 GitHub。

自动刷新间隔由 `intervalMinutes` 控制。也可以在看板“设置”里的“自动刷新间隔”选择常用分钟数，或在“自定义分钟”里输入任意分钟数；修改后后台服务会按新间隔继续运行。
现在“自动刷新”和“记录快照”是同一个功能：开启自动刷新的活动会定时拉取接口、记录小时快照，并在成功后自动推送公开看板。
已经打开的看板页面会自动检查当前活动的数据截止时间；发现后台服务或公网发布的数据变新后，会自动重新载入页面，不需要手动刷新。页面自动重载只读取已生成的数据，不会额外请求美团接口。
全量活动列表同步是固定每天 8:00-22:00 每隔 2 小时执行一次；它会更新活动 ID、标题、状态和活动时间，并自动把“进行中”的活动加入“自动刷新”名单。

每次刷新成功后，会写入小时快照：

```text
data/hourly-snapshots-活动ID.json
```

本机新增活动会保存到：

```text
data/saved-activities.json
```

这个文件保存活动 ID、标题、活动时间、规则图片路径、是否开启自动刷新等公开元信息；不会保存 Cookie、请求头或 mtgsig。自动推送开启时，保存的活动也会同步到公网看板，公网用户可以切换活动。

飞书活动日报在本地看板右上角“设置”里的“飞书通知”配置：

- `Webhook URL`：粘贴飞书机器人 Webhook。
- `签名密钥`：如果飞书机器人开启了签名校验，就填写签名密钥；未开启可留空。
- `通知时间`：用英文逗号分隔，例如 `12:00,22:00`。

Webhook 和签名密钥保存到 macOS 钥匙串，不写入仓库、HTML 或日志。只有在“切换”活动列表里开启了“飞书通知”的活动才会发送日报。飞书接口返回失败时，服务不会把该时间点标记为成功，会按 10 分钟、30 分钟、60 分钟最多重试 3 次；如果飞书返回频率限制，需要减少测试发送或等待限制解除。

macOS 已配置 `launchd` 常驻任务：

```text
~/Library/LaunchAgents/com.tanwenjie.meituan-dashboard.plist
```

只要电脑开机并登录，该服务会在后台运行；关闭看板网页或终端窗口不会影响刷新。
如果电脑异常关机，重新开机并登录系统后，`launchd` 会自动尝试启动该服务。若看板数据没有继续更新，双击桌面的“美团看板服务.command”，选择“查看服务状态”或“重启服务”即可恢复。

日志文件：

```text
logs/background-refresh.log
logs/background-refresh-error.log
```

## 桌面入口

后台服务控制入口：

```text
~/Desktop/美团看板服务.command
```

双击后可以选择查看状态、启动服务、停止服务、重启服务，或打开本地看板。
日常只保留这个入口即可：先确认服务运行，再选择“打开本地看板”。

业务与用户数据看板也已配置独立后台服务：

```text
~/Desktop/业务用户看板服务.command
```

这个入口用于控制 `http://127.0.0.1:8791/` 的统一看板入口：

- “查看服务状态”：查看开机自启服务是否正在运行。
- “启动服务 / 停止服务 / 重启服务”：手动控制本地接口代理。
- “打开本地看板”：打开业务用户实时看板。
- “打开公网看板”：打开 GitHub Pages 上的业务异常监控公开快照。
- “写入/更新钥匙串账号”：把中台账号密码写入 macOS 钥匙串，服务启动时从钥匙串读取，不写入 HTML、plist 或日志。

手机端如果和这台 Mac 在同一个 Wi-Fi 下，优先使用局域网地址访问业务看板，例如 `http://192.168.1.37:8791/#business`。这个地址直接读取本机服务和本机公开快照，不经过 GitHub Pages，不需要翻墙，速度也更稳定。局域网 IP 可能会随 Wi-Fi 变化而变化；服务状态里显示的“同一 Wi-Fi 手机可访问”地址为准。

对应的 macOS 开机自启任务是：

```text
~/Library/LaunchAgents/com.tanwenjie.business-user-dashboard.plist
```

日志文件：

```text
logs/business-user-dashboard.log
logs/business-user-dashboard-error.log
```

如果重启电脑后看板打不开，双击桌面的“业务用户看板服务.command”，先选择“查看服务状态”；如果未启动或日志报错，再选择“重启服务”。如果提示缺少钥匙串账号，选择“写入/更新钥匙串账号”重新录入中台账号密码。

## 开放平台毛利看板

毛利看板只允许从本机访问：公网统一入口不显示该看板，公网直接访问毛利页面会返回普通经营看板；同时不发布原始 Excel 或明细 JSON。文件位置：

```text
data/private/开放平台毛利表2026.xlsx
data/private/open-platform-profit-workbook.json
data/open-platform-profit-dashboard.json
```

上述目录和数据文件已加入 `.gitignore`，不会被 Git 提交。更新流程是：先从飞书邮箱下载蒋敏最新发送的 `开放平台毛利表2026.xlsx`，再运行项目脚本生成看板数据。macOS 终端在项目根目录执行：

```bash
zsh scripts/refresh-open-platform-profit-dashboard.zsh
```

正常结果会显示“毛利看板数据已更新”。更新后刷新本地统一入口的“开放平台毛利”页面即可。若提示没有找到 Excel，先确认附件已下载到 `/Users/tanwenjie/Downloads/开放平台毛利表2026.xlsx`；若浏览器页面打不开，先用桌面的“业务用户看板服务.command”重启本机服务。

业务异常监控看板目前包含：

- 指标卡：默认展示全部业务、全部订单数、全部 GMV、异常业务 4 个总览指标。
- 业务列表：默认固定展示今日全部业务，可按订单或佣金筛选，支持按当前订单、变化比例、影响量排序，并支持严重、重点、轻微、上升、数据缺失等异常类型。
- 业务用户刷新开关：业务订单仍统一按 10 分钟更新；小时趋势默认收起，展开后才读取。业务行的“用户高频更新”只控制该业务用户前 100 名是否每 10 分钟更新。该勾选按用户明细业务 ID 独立保存，设置中心保存其他参数时不会覆盖。关闭后，该业务用户明细和用户历史只在设置中心配置的固定时间更新，默认 `12:00、17:00、22:00`，支持继续增加时间。
- 页面自动检查：默认按设置频率读取一次本机缓存，页面重新获得焦点时也会立即检查；这两种检查都不请求中台。服务端会把所有未带“手动刷新”标记的页面请求强制改为只读缓存，因此旧页面、重复标签页或旧版前端也不能触发中台全量刷新。业务订单和当前趋势每 10 分钟最多请求一次中台，手动刷新除外。
- 重点用户：统一入口左侧提供独立“重点用户看板”。添加时只输入一次用户 ID，系统按用户 ID 全局唯一保存，不再为每个业务重复添加。默认“按业务看”并选择美团外卖节，可切换其他业务查看全部重点用户；也可切换“按用户看”，查看某位用户所有有单业务。在“按业务看”中点击用户整行或手机端用户卡片，会直接弹出该用户全部有单业务数据，内容与“按用户看”一致；复制、改名、备注、置顶、移除、每日订单展开等行内操作不会误触弹窗。修改名称、观察备注、置顶和移除均为用户级操作，在所有业务视图保持一致。新增提交后立即显示占位用户，后台一次性扫描全部业务，按用户 ID 精确补全姓名、明文手机号和有单关系；之后每 10 分钟只刷新已经发现的有单业务。近7天、本月、近30天和自定义日期只改变右侧每日订单列。公网在同一台运行本地服务的电脑上添加时会同步本地配置；手机或其他电脑无法直接写入这台 Mac，新增内容先保存在该浏览器的 `localStorage`。
- 跨业务用户查询：业务列表筛选区提供独立的“查询用户 ID 或名称”入口，不与业务名称搜索混用。查询后在弹窗内展示该用户所有有单业务的今日订单、昨日同时刻、变化、状态和每日订单，支持近7天、本月、近30天切换；重名时先选择具体用户。普通查询和日期切换只读取已保存缓存，不请求中台。本地只有明确点击“刷新该用户”时才逐业务精确查询，当前约为每个业务 1 次；公网首次使用该功能时按需加载独立加密用户关系索引和相关业务分片，不影响看板首页加载速度。
- 重点用户电脑端高度：重点用户数量较少时，白色表格区域仍会自动占满浏览器剩余高度，不再让面板随最后一行提前结束；用户较多时继续在表格区域内纵向滚动，业务列表和手机端布局不受影响。
- 业务历史订单：点击业务列表中的业务名称，会在当前页面弹窗展示该业务按日订单；支持近七天、上周、本月和自定义日期，不再离开业务列表。
- 重点用户观察备注与置顶：备注只在重点用户看板展示，点击笔图标可按行新增或删除；姓名旁的向上箭头可置顶或取消置顶，支持多个置顶用户，最近置顶的排在最前。本地写入成功后页面立即更新，不再等待整份看板重新发布；备注和置顶状态共用小型加密增量快照，公网页面每15秒检查一次，通常保存后约10～30秒可见，且不会请求中台接口。公网可以直接操作；同一台运行本地服务的电脑会同步本机配置，其他设备先保存在当前浏览器。
- 宽表格滚动：业务、用户、每日订单和弹窗表格统一使用加粗的横向滚动条，日期或字段较多时可在表格底部直接拖动。
- 前 100 变化：开启用户高频更新的业务每 10 分钟重新获取当时的今日订单前 100。新进入者从该轮开始进入高频更新并标记进入时间；离开者停止高频更新，保留最后一次值，直到下一个固定全量时段或手动刷新。前 100 只请求当前订单，对比基准优先读取历史快照。
- 业务详情：点击业务行的“查看用户”，固定按今日对比昨日同时刻判断流失风险，也可切换增长用户和全部用户；增长用户包含所有同时刻订单净增长大于0的用户，并默认按增长影响量降序。手机端使用全屏数据矩阵，固定用户名称列，指标区可横向滑动；点击用户行可展开手机号、版本和所选范围的每日准确值。为保证两千人以上列表的响应速度，手机端首批渲染80人并支持继续加载，但人数和合计始终按完整筛选结果计算。电脑端继续使用原宽表格。日期只改变每日订单列，不改变今日异常判断；点击“刷新用户”时保留旧列表，完整加载成功后一次替换。
- 用户异常维度：支持今日、近7天和本月。今日对比昨日同时刻；近7天对比前7天；本月截至今天对比上月相同日期进度。缺少基准时显示“-”且不纳入异常，10%以内波动按正常处理。
- 用户流失分析：主表固定显示今日订单、昨日同时刻、上周同时刻、变化比例、影响订单、状态和所选日期范围内的每日订单；删除独立原因列和历史订单页，让异常判断与每日走势在同一张表中核对。
- 用户数据一致性：业务列表和今日用户明细统一使用中台 `order_valid`（有效订单）口径。单日接口可能为同一用户ID返回多条业务记录，必须按用户ID求和；2026年7月14日实测美团外卖节原始1,088条记录求和为75,898单，与业务列表75,898单完全一致。今日列表把已勾选业务的最新前100用户合并到单日全量缓存；右侧每日订单来自约65天历史缓存。打开的单日全量数据比业务列表落后9分钟以上时，本地会保留旧数据并自动刷新当前业务。公网和本地使用同一份加密发布缓存，并分别显示今日数据时间与每日历史数据时间。
- 用户请求量：页面每1分钟检查缓存、页面重新获得焦点时检查缓存都不会请求中台。每10分钟业务汇总、目录、每日数据、默认趋势和快照约38次；当前15个高频业务前100约150次；8位重点用户近期每轮约84～114条已发现业务关系，即合计约272～302次/轮、约3.9～4.35万次/天。新增重点用户时额外一次性扫描当前全部业务，扫描完成后不再每轮请求无单业务。完整明文手机号索引每天最多更新一次，当前25,043位用户约835次。固定时段的65天历史由一次日期范围返回65个日期列，再按业务用户分页；2026年7月14日12:00实测全业务一轮约4,000次，三个固定时段约1.2万次/天。查看用户命中且单日全量未过期时为0次；打开落后9分钟以上的当前业务或手动刷新，只分页请求该业务当天有效订单。
- 公网移动端：公开首页只包含业务汇总和用户分片清单；点击某个业务后再加载该业务的加密用户分片，避免手机一次解密约30MB全量用户历史。旧版手机浏览器使用本地 `fflate` 兼容 gzip 解压，用户数据仍保持密码加密。
- 手机端业务与重点用户：宽度不超过 960px 时，业务列表和重点用户列表改为纵向数据卡片，直接展示当前值、同期对比、变化比例、影响量和状态，并保留历史订单、查看用户、每日订单、备注、置顶和移除等操作；卡片与电脑端表格读取同一份页面状态，电脑端继续使用原宽表格。
- 手机端每日业务数据：使用“业务列固定、日期区域横向滑动”的紧凑矩阵，默认自动定位到最新日期；支持“按日期看”和“按业务看”，后者可逐项展开查看所选范围内的全部每日准确值。矩阵与电脑端原表格直接读取同一个 `businessDaily` 数据对象，日期合计按当前筛选结果实时求和。
- 手机端页头：业务异常监控和重点用户页面的说明、更新时间及辅助按钮默认收起为一行，点击“展开页头”可恢复完整信息；收起状态只在手机端生效，电脑端布局不变。按钮保持至少 44×44 像素点击区域；浏览器禁止本地存储时仍可正常展开和进入页面。
- 业务下钻：点击业务名称会在当前页面弹出该业务历史订单；点击业务行其他区域或“查看用户”进入用户异常列表。历史订单和每日业务数据在日期超过7天时使用宽表及常驻横向拖动条，避免日期列被压缩或截断。
- 每日日期切换：近7天、本月、近30天和自定义日期都从已保存的约65天历史中切片，不重复请求中台；日期输入框限制在当前已缓存范围内，避免把缺失日期误显示为0。
- 用户备注名：本地可在用户名称旁点击“修改”保存统一备注，按用户ID应用到用户列表、历史订单和重点用户；留空保存会删除备注并恢复接口原名。备注保存在 `data/business-user-aliases.json`，不提交明文文件，保存后会进入公网加密快照。公网也可修改备注，覆盖值只保存在当前浏览器的 `localStorage` 中。
- 重点用户公网同步：本地新增或移除重点用户后，服务会直接使用现有缓存重新生成并推送公网加密快照，不刷新全量用户，发布时只校验默认业务的3条趋势；GitHub Pages 更新后公网重点用户与本地一致。
- 重点用户兼容迁移：旧版按“业务 + 用户”保存的重复记录会在首次读取时按用户 ID 合并；名称、手机号、版本、备注、置顶和最早添加时间保留，原文件同时备份为 `data/business-focus-users.pre-global-backup.json`。重点用户当前值只使用本轮精确查询或当天日期列，历史缓存中的旧 `todayOrders` 不允许替代今日订单。
- 用户每日订单：不再单独跳转历史订单页。今日用户表右侧直接展示每日订单，默认近7天，并支持本月、近30天和自定义日期；用户行的“+”可直接加入重点用户看板，用户ID可点击复制，搜索框可一键清空。
- 用户数据校验：外层业务订单和用户明细都使用有效订单口径；分页产生的完全重复用户行不重复相加。弹窗把业务净下降拆成“下降用户、增长抵消、用户净下降、未定位差额”，并提供流失风险、增长用户和全部用户三类列表。用户快照优先保存覆盖全部用户的缓存，再用时间更新的前100数据覆盖；旧快照无法追溯补齐，新的完整同时刻归因需要持续记录至少一天。
- 业务小时趋势：指标区下方展示单业务 24 小时订单折线图，包含今日、昨日、上周同期三条线，默认选中“美团外卖节”，可在趋势图右上方下拉切换业务；该图目前来自中台小时趋势接口，不是半小时快照推导。公网加密快照默认包含“美团外卖节”的趋势数据。
- 每日业务数据日期：日期范围只影响“每日业务数据”视图，切换到该视图后才显示日期控件；选完日期或快捷范围后需要点击“查询”才刷新。
- 飞书通知配置：从本地页面保存 Webhook 时会走本机服务接口，服务已支持 `file://` 页面的 CORS 预检；测试失败时页面会显示后端返回的具体失败原因。设置中心会列出启动预热失败、快照记录失败、接口数据缺失、业务为空、全部订单为 0、大量业务数据不变化和公网发布失败 7 个实际通知项目，可逐项勾选；同类项目 1 小时内最多通知一次。
- 业务完整性：主列表以中台“业务列表”接口作为底表，再合并“业务统计”接口的订单、佣金、成交金额；因此今日 0 单但仍在中台业务库里的业务也会展示。
- 启动预热：本地服务启动后会加载已有用户明细、历史和手机号索引，并刷新一次今日业务；只对开启“用户高频更新”的业务刷新前 100，不再启动时全量请求所有业务用户。
- 用户手机号：业务下钻接口默认返回脱敏手机号；本地服务每天最多一次分页读取中台完整用户列表，建立推广 ID 到明文手机号的索引并保存到 `data/user-phone-index.json`。用户列表、历史订单和重点用户都统一从该索引补全；无法取得合法11位手机号时显示“-”，不再展示脱敏号码。
- 设置中心：只在本地看板展示，只包含配置，不展示业务数据；可配置异常阈值、页面自动检查频率、快照频率、中台账号密码、飞书 Webhook、签名密钥和通知项目。每次打开都会重新读取已保存配置，普通选项和通知开关恢复真实值；账号、密码、Webhook 和签名密钥在输入框中显示“已保存”掩码，避免把钥匙串中的真实敏感值返回网页。输入新值可替换，保持掩码直接保存不会覆盖原值；只更新账号或密码其中一项时，另一项沿用钥匙串原值并一起完成登录测试。页面自动检查只读取本机缓存，不会额外请求中台；中台账号密码保存到 macOS 钥匙串，保存时会先登录测试，测试失败不会覆盖当前账号。账号密码能够成功登录只代表可以取得访问令牌；中台账号权限、接口状态、网络、限流或接口字段变化仍可能导致部分数据拿不到，服务会通过“接口数据缺失”或“快照记录失败”通知暴露这类问题。
- 飞书测试：配置 Webhook 和签名密钥后，可点击“发送测试消息”确认机器人能正常收到消息。
- 主看板缓存：后台会把每个日期范围最近一次成功返回的主看板数据保存到本地；页面首次打开优先显示缓存，点击右上角刷新才强制请求最新中台数据。
- 对比基准：当前页面先固定展示今天，业务异常、流失风险、增长用户、全部用户和重点用户都优先按完全相同分钟槽判断。精确分钟缺失时，使用目标时刻前后20分钟内最近的一条快照；超过20分钟仍无数据才显示“-”。业务列表和用户列表都把当前筛选结果的多数数据时间、昨日多数参考时刻和上周多数参考时刻放在对应表头，行内只标记与多数时刻不同的数据，避免相同时间重复占用每一行。用户列表按每行数据自己的更新时间选择目标分钟，且基准快照必须同时包含该业务和该用户。完整当日用户接口未返回的历史用户按该批次0单处理，数据时间沿用完整接口批次，不再错误沿用早先历史缓存时间。新快照只保存本轮确实在该分钟更新的用户，并带有严格用户快照标记；升级前无法证明逐用户更新时间的旧快照只作为明确标注的“历史参考”。订单和佣金共用同一份快照，不使用整点小时趋势或昨日全天替代。上周同时刻和近7日同期均值也使用同样的精确优先、邻近兜底规则。
- 快照记录：后台服务按设置的间隔记录业务和用户快照，当前设置为每10分钟，并严格对齐每天自然时间槽 `00:00、00:10、00:20...`。快照只保留今天和前7个自然日，共8天；每天首次记录后会自动删除更早快照，并把用户快照裁剪为订单和佣金，姓名、手机号、版本等资料不重复保存。近30天、65天等每日历史订单使用独立缓存，不会随快照清理。页面“刷新”只请求最新数据，不记录快照；电脑关机或服务未运行时，错过的时间槽不会伪造补齐。设置中心的“立即记录快照”会写成手动快照，不占用固定10分钟槽。快照调度使用版本锁防止快速保存配置时产生多个定时器，所有写入串行执行；读取时会跳过不完整行，单条快照损坏不会再阻断整个看板刷新。
- 公网快照：每次定时快照成功后，会生成 gzip 压缩后的加密公开数据 `data/business-dashboard-public.enc.json`，并自动推送到 GitHub Pages。主快照使用 60,000 次密钥派生，降低手机微信里的解密等待；公网首屏优先使用浏览器缓存和本地缓存快速展示，再后台读取最新加密快照。公网浏览器首次输入密码并解密成功后，会把最近一次公开快照缓存在该浏览器本地 7 天。
- 公网访问保护：GitHub Pages 是静态站点，不能做真正的服务端登录拦截；本项目改为加密公开数据文件。公网打开页面后需要输入访问密码，浏览器本地缓存 7 天，7 天内不需要重复输入；浏览器禁止本地存储时改用当前页面内存传递密码，不会因存储异常卡在空白页。
- 公网用户明细：加密快照里包含业务列表、已缓存用户明细、用户历史和重点用户。生成公网文件时只读取本地缓存，不再为了发布重复全量请求中台；公网不会实时请求本机接口。
- 公网跨业务用户索引：`data/business-global-user-index.enc.json` 只保存用户与业务的加密关系索引，公网执行用户 ID 或名称查询时才加载；数据仍需使用看板访问密码解密，不发布中台账号、请求标头或本地明文缓存。
- 请求统计：服务按日期、接口路径和用途记录实际中台请求次数，设置中心可查看今日总请求和业务用户接口请求。统计文件 `data/business-api-request-stats.json` 只保存在本地，不提交公开仓库。
- 快照异常通知：如果快照记录失败、接口数据缺失、业务为空、订单全为 0，或大量活跃业务数据与上一条快照完全一致且重点业务也未变化，会通过飞书机器人通知；重点业务按“美团外卖节、闪购、当前订单量前 5”判断，同类问题 1 小时内最多通知一次。

## 手动输入保存

页面“预估数据”列的手动输入会保存两份：

- 浏览器本地保存：用于页面立即刷新后继续保留。
- 后台服务文件保存：写入 `data/manual-overrides.json`，用于浏览器重启或后续重新打开看板后恢复。

清空某一天输入框后，会恢复使用接口默认值，并同步清除后台保存文件里的对应日期。

## 新增或切换活动

最简单流程：

1. 打开看板右上角“设置”。
2. 在“新增活动 ID”里填写新的活动 ID。
3. 如果这个活动需要长期跟踪，勾选“自动刷新”；如果只是临时查看，不要勾选。
4. 点击“新增”。
5. 回到页面右上角点击“刷新”。

只有开启“自动刷新”的活动，才会写入 `data/watch-config.json` 的 `activityIds`，后续按设置间隔继续刷新并记录快照。活动列表同步会自动为“进行中”活动开启自动刷新；未开始或已结束活动如果没有手动开启，则只会在你手动点击“刷新”时更新当前数据。

“切换”弹窗里的“活动管理”列表用于日常控制活动：

- 搜索框：可以按活动 ID 或标题关键词过滤活动。
- 分组：按“进行中、未开始、已结束”展示，默认打开“进行中”。
- “全开 / 全关”：一键开启或关闭当前分组和当前搜索结果里的活动自动刷新。
- “自动刷新”：控制该活动是否进入自动刷新和快照记录名单。
- “默认展示”：控制打开看板时默认展示哪个活动。
- “上次刷新时间”：显示该活动最近一次接口刷新写入看板的时间。
- “切换”：只切换当前查看的活动，不修改默认展示。
- “刷新”：立即刷新这一行对应的活动。
- “删除”：二次确认后删除活动、规则图和快照文件。

新增活动时页面会先显示“活动 ID”的占位标题；点击刷新成功后，标题、活动时间、档位和规则图会用接口返回的数据自动替换。

## 今日预估口径

自动刷新会记录半小时快照，并写入：

```text
data/hourly-snapshots-活动ID.json
```

看板顶部的“今日预估 GMV / 今日预估订单数”会显示预估来源。优先级为：手动输入 > 前一日完整快照 > 前两日完整快照 > 上周同日完整快照 > 最近可用完整日快照 > 历史完整日平均值 > 当前接口值。凌晨当天接口还没返回数据时，页面会暂时显示最新有数据的日期，并在标签里写明日期。

## 对外访问

推荐用 GitHub Pages 对外展示静态看板。后台服务刷新后会尝试执行：

```bash
git add index.html meituan-dashboard-preview.html README.md .gitignore .env.example scripts assets data
git commit -m "Update Meituan dashboard ..."
git push origin main
```

注意：GitHub 自动推送需要本机命令行 GitHub 凭证可用。网页里登录 GitHub 不等于命令行 `git push` 已登录。
当前配置已开启自动推送：本机后台服务刷新成功后，会自动提交公开文件并推送到 GitHub，GitHub Pages 构建完成后，公网看板会自动显示最新发布数据。

公开 GitHub Pages 上的“刷新”只会重新加载已发布到 GitHub 的最新静态数据；真正请求美团接口的刷新只在本机后台服务里执行。不要把 Cookie、请求标头或 mtgsig 放到公开网页里。
公网访问时只保留“刷新”和“切换”；新增活动、自动刷新、默认展示、删除等配置只在本机看板开放。
公网自动更新不依赖 Codex 对话窗口，只依赖这台 Mac 的后台服务、网络、GitHub 凭证和 GitHub Pages。关闭本对话不影响后台服务；如果电脑关机、睡眠、断网或后台服务停止，公网就不会继续拿到新推送。

## 安全注意

`.env` 中包含 Cookie 和 mtgsig，不能上传到 GitHub，也不能发给别人。
当前公开仓库不会提交 `.env`、完整请求标头、手动输入保存文件或日志；但看板页面中的活动标题、活动规则图片、每日订单量、GMV、快照趋势等业务数据会公开展示。
